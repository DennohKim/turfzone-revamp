# Turfzone Invariants

These are the backend rules that must remain true across future refactors.

## Booking

- No two active bookings overlap on the same field.
- Active means `Confirmed` or `PendingPayment` with a non-expired hold.
- Pending holds expire after the configured hold window, currently 7 minutes.
- Booking amount must be positive.
- Booking end time must be after booking start time.
- Commission plus manager share must equal booking amount.
- Booking confirmation must be driven by Paystack webhook/verify or wallet debit, never by client-reported success.

## Availability

- Availability is computed from opening hours minus full-day/partial exceptions minus active bookings.
- Expired holds do not block availability.
- Slot generation must respect each field's `slot_minutes` granularity.

## Wallet

- Wallet ledger is append-only.
- Wallet balance must equal the sum of ledger entries.
- Duplicate ledger references must not mutate balance.
- Wallet debits cannot overdraw the balance.
- Eligible M-Pesa booking refunds default to wallet credit in MVP.

## Payments

- Paystack webhook signature must be verified with HMAC-SHA512 over the raw body.
- Webhook handling must be idempotent by booking/payment reference.
- Payment amount and currency must match the server-created booking/payment record.
- Public pricing is KES-only for MVP.
- Paystack secret key must never be exposed to the frontend.

## Manager Settlement

- A verified manager must have exactly one payout destination: bank account or M-Pesa wallet.
- Manager payout/subaccount payloads must be generated by provider code, not route handlers.
- Provider payout references are idempotency keys.

## Notifications

- Notification sending is asynchronous.
- Notification idempotency keys must prevent duplicate booking confirmations/reminders.
- Failed notifications must not block booking/payment state transitions.
